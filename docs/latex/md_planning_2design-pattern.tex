 Este documento describe la arquitectura de software de la librería \doxylink{namespacedirac__solver}{dirac\+\_\+solver}. El diseño se basa en un conjunto de patrones de diseño canónicos para garantizar que el código sea modular, extensible, mantenible y fácil de usar. El objetivo principal es lograr una clara separación de conceptos (Separation of Concerns)\+: la definición del problema físico se mantiene desacoplada de los algoritmos numéricos utilizados para resolverlo. \hypertarget{md_planning_2design-pattern_autotoc_md15}{}\doxysection{\texorpdfstring{2. Patrón Builder -\/ Configuración de la Simulación}{2. Patrón Builder - Configuración de la Simulación}}\label{md_planning_2design-pattern_autotoc_md15}
Propósito\+: Construir un objeto complejo (Simulation\+Problem) paso a paso, proporcionando una API fluida y legible que separa la construcción de la representación.

Aplicación\+: El usuario final define un problema completo ensamblando sus partes constituyentes (geometría, potencial, condiciones, etc.) sin necesidad de un constructor con múltiples y confusos argumentos. Un Dirac\+Problem\+Builder es el punto de entrada principal para el usuario. Este objeto guía la creación de una simulación válida y cohesiva.

\hypertarget{md_planning_2design-pattern_autotoc_md18}{}\doxysection{\texorpdfstring{3. Patrón Strategy -\/ Flexibilidad en los Algoritmos}{3. Patrón Strategy - Flexibilidad en los Algoritmos}}\label{md_planning_2design-pattern_autotoc_md18}
Propósito\+: Definir una familia de algoritmos, encapsular cada uno y hacerlos intercambiables. Permite que el algoritmo varíe independientemente del cliente que lo utiliza.

Aplicación\+: Este es el patrón más importante para el núcleo numérico en solvers/. Permite al usuario cambiar el método de integración temporal (ej. Crank\+Nicolson vs. Split\+Operator) o las condiciones de frontera (ej. Dirichlet vs. Absorbing) sin modificar el código del bucle de simulación principal.

\hypertarget{md_planning_2design-pattern_autotoc_md20}{}\doxysection{\texorpdfstring{4. Patrones Template y Factory Method -\/ Extensibilidad de Componentes Físicos}{4. Patrones Template y Factory Method - Extensibilidad de Componentes Físicos}}\label{md_planning_2design-pattern_autotoc_md20}
Propósito\+:


\begin{DoxyItemize}
\item Template Method\+: Define el esqueleto de un algoritmo en una clase base, delegando la implementación de ciertos pasos a las subclases.
\item Factory Method\+: Proporciona una interfaz para crear objetos, pero permite que las subclases alteren el tipo de objetos que se crearán.
\end{DoxyItemize}

Aplicación\+: Se combinan en el módulo potentials/ (y análogamente en geometry/ y conditions/).


\begin{DoxyItemize}
\item potentials/base.\+py define una Base\+Potential (el Template) que requiere que todas las subclases implementen un método evaluate().
\item Una función create\+\_\+potential() (el Factory Method) centraliza la lógica de instanciación, permitiendo crear potenciales a partir de un string (ej. desde un archivo de configuración).
\end{DoxyItemize}

\hypertarget{md_planning_2design-pattern_autotoc_md22}{}\doxysection{\texorpdfstring{5. Patrón Observer -\/ Visualización y Monitoreo}{5. Patrón Observer - Visualización y Monitoreo}}\label{md_planning_2design-pattern_autotoc_md22}
Propósito\+: Definir una dependencia uno-\/a-\/muchos entre objetos, de modo que cuando un objeto (Subject) cambia de estado, todos sus dependientes (Observers) son notificados y actualizados automáticamente.

Aplicación\+: Desacopla el motor de la simulación de los componentes de visualización y registro de datos. El objeto Simulation (Subject) no necesita saber nada sobre cómo se muestran los datos. Simplemente notifica a sus observadores (ej. Wavefunction\+Plotter, Energy\+Logger) en cada paso de tiempo.

\hypertarget{md_planning_2design-pattern_autotoc_md24}{}\doxysection{\texorpdfstring{6. Patrón Facade -\/ Interfaz Simplificada de I/O}{6. Patrón Facade - Interfaz Simplificada de I/O}}\label{md_planning_2design-pattern_autotoc_md24}
Propósito\+: Proporcionar una interfaz unificada y simplificada a un conjunto de interfaces en un subsistema.

Aplicación\+: El módulo io/ puede contener lógica compleja para guardar en diferentes formatos (HDF5, npz), cargar, y parsear archivos de configuración. El patrón Facade oculta esta complejidad detrás de una única clase IOManager con métodos sencillos como save(simulation, path) y load(path).\hypertarget{md_planning_2design-pattern_autotoc_md25}{}\doxysection{\texorpdfstring{7. Patrón Decorator – Extensión Dinámica de Funcionalidades}{7. Patrón Decorator – Extensión Dinámica de Funcionalidades}}\label{md_planning_2design-pattern_autotoc_md25}
{\bfseries{Propósito\+:}} Añadir responsabilidades a un objeto de manera dinámica sin modificar su estructura.

{\bfseries{Aplicación\+:}} Permite extender solvers o visualizadores sin cambiar su código base. Por ejemplo\+:


\begin{DoxyItemize}
\item Medir tiempos de ejecución de cualquier solver.
\item Añadir logging detallado a un {\ttfamily \doxylink{classPotential}{Potential}} o {\ttfamily Solver} ya existente.
\item Incorporar validaciones numéricas (ej. estabilidad de la norma de la función de onda).
\end{DoxyItemize}

Esto permite una instrumentación ligera y flexible, útil para depuración y optimización.

\hypertarget{md_planning_2design-pattern_autotoc_md27}{}\doxysection{\texorpdfstring{8. Patrón Adapter – Interoperabilidad con Librerías Externas}{8. Patrón Adapter – Interoperabilidad con Librerías Externas}}\label{md_planning_2design-pattern_autotoc_md27}
{\bfseries{Propósito\+:}} Convertir la interfaz de una clase existente en otra que el cliente espera, permitiendo la colaboración entre clases incompatibles.

{\bfseries{Aplicación\+:}} El solver puede necesitar interoperar con {\bfseries{bibliotecas externas}} (Sci\+Py, PETSc, Cu\+Py, Py\+Torch). Cada una tiene interfaces distintas para vectores, matrices y rutinas de eigenvalores.

El Adapter proporciona una capa intermedia que traduce entre la API de {\ttfamily \doxylink{namespacedirac__solver}{dirac\+\_\+solver}} y la API de la librería de backend. Así, el código cliente no depende de la librería específica.

Ejemplo hipotético\+:


\begin{DoxyCode}{0}

\DoxyCodeLine{\ \ \ \ \textcolor{keyword}{def\ }\_\_init\_\_(self,\ petsc\_solver):}

\DoxyCodeLine{\ \ \ \ }
\DoxyCodeLine{\ \ \ \ \textcolor{keyword}{def\ }solve(self,\ A,\ b):}

\end{DoxyCode}


Esto facilita ofrecer múltiples backends (Num\+Py, Sci\+Py, GPU) sin que el usuario final deba cambiar su código.

\hypertarget{md_planning_2design-pattern_autotoc_md29}{}\doxysection{\texorpdfstring{9. Relaciones entre los Patrones}{9. Relaciones entre los Patrones}}\label{md_planning_2design-pattern_autotoc_md29}
Los patrones no funcionan de forma aislada; se complementan para mantener la arquitectura coherente\+:


\begin{DoxyItemize}
\item {\bfseries{Builder → Strategy / Template-\/\+Factory}}\+: El {\ttfamily Dirac\+Problem\+Builder} ensambla la simulación seleccionando estrategias numéricas y componentes físicos generados mediante fábricas.
\item {\bfseries{Strategy ↔ Observer}}\+: Cada estrategia de integración puede notificar cambios en la simulación que luego son procesados por observadores como {\ttfamily Energy\+Logger}.
\item {\bfseries{Observer ↔ Decorator}}\+: Los observadores mismos pueden ser decorados para añadir validaciones o mediciones de rendimiento sin tocar su código base.
\item {\bfseries{Facade ↔ Adapter}}\+: El {\ttfamily IOManager} (Facade) puede usar adaptadores internos para guardar datos en distintos backends (HDF5, Num\+Py, etc.).
\item {\bfseries{Decorator ↔ Strategy}}\+: Un {\ttfamily Solver} puede envolverse con decoradores para medir tiempos de ejecución de cada algoritmo concreto.
\end{DoxyItemize}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Constructores telescópicos con muchos parámetros.
\item {\bfseries{Descartado porque\+:}} Generan código ilegible, difícil de mantener y propenso a errores.
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md33}{}\doxysubsection{\texorpdfstring{2. Strategy}{2. Strategy}}\label{md_planning_2design-pattern_autotoc_md33}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Uso de {\itshape flags} o {\ttfamily if/else} en el bucle principal.
\item {\bfseries{Descartado porque\+:}} Acopla la lógica de alto nivel con algoritmos específicos, reduce extensibilidad.
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md34}{}\doxysubsection{\texorpdfstring{3. Template + Factory Method}{3. Template + Factory Method}}\label{md_planning_2design-pattern_autotoc_md34}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Clases concretas instanciadas directamente por el cliente.
\item {\bfseries{Descartado porque\+:}} Rompe la consistencia (el usuario tendría que conocer detalles internos para instanciar).
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md35}{}\doxysubsection{\texorpdfstring{4. Observer}{4. Observer}}\label{md_planning_2design-pattern_autotoc_md35}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Llamadas directas a funciones de visualización/logging dentro del bucle principal.
\item {\bfseries{Descartado porque\+:}} Acopla la simulación con la visualización, volviendo el código rígido.
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md36}{}\doxysubsection{\texorpdfstring{5. Facade}{5. Facade}}\label{md_planning_2design-pattern_autotoc_md36}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Acceso directo a módulos {\ttfamily saver}, {\ttfamily loader}, {\ttfamily parser}.
\item {\bfseries{Descartado porque\+:}} El usuario tendría que conocer la complejidad interna de I/O, aumentando la curva de aprendizaje.
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md37}{}\doxysubsection{\texorpdfstring{6. Decorator}{6. Decorator}}\label{md_planning_2design-pattern_autotoc_md37}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Herencia múltiple para añadir funcionalidades adicionales.
\item {\bfseries{Descartado porque\+:}} La herencia rígida genera duplicación de código y combinaciones explosivas (ej. {\ttfamily Logging\+Solver\+With\+Validation}).
\end{DoxyItemize}\hypertarget{md_planning_2design-pattern_autotoc_md38}{}\doxysubsection{\texorpdfstring{7. Adapter}{7. Adapter}}\label{md_planning_2design-pattern_autotoc_md38}

\begin{DoxyItemize}
\item {\bfseries{Alternativa\+:}} Reescribir código para cada librería externa (Sci\+Py, PETSc, Cu\+Py).
\item {\bfseries{Descartado porque\+:}} Rompe portabilidad, obliga a duplicar código cliente y reduce mantenibilidad.
\end{DoxyItemize}

\hypertarget{md_planning_2design-pattern_autotoc_md40}{}\doxysection{\texorpdfstring{11. Conclusión}{11. Conclusión}}\label{md_planning_2design-pattern_autotoc_md40}
El uso de patrones de diseño en {\ttfamily \doxylink{namespacedirac__solver}{dirac\+\_\+solver}} asegura\+:


\begin{DoxyItemize}
\item {\bfseries{Modularidad\+:}} cada componente cumple una responsabilidad clara.
\item {\bfseries{Extensibilidad\+:}} fácil añadir nuevos algoritmos, potenciales o backends.
\item {\bfseries{Mantenibilidad\+:}} menor duplicación de código y menor acoplamiento.
\item {\bfseries{Flexibilidad\+:}} integración fluida con librerías externas y herramientas de depuración.
\end{DoxyItemize}

Esto permite que la librería crezca de manera ordenada y soporte tanto investigación académica como aplicaciones en entornos de alto rendimiento.
